What happens when an LLM returns garbage

OTP: let it crash Node: defensive coding
1

LLM returns garbage JSON

Model hallucinates invalid syntax, or returns a response in an unexpected format. This is normal — LLMs are non-deterministic.

2

GenServer crashes

The process exits. Its heap is reclaimed. No shared state is corrupted — the crash is contained to this one process.

3

Supervisor detects + restarts

DynamicSupervisor sees the child exit, applies the restart strategy. New GenServer starts in a clean state. Automatic, declarative, zero application code.

4

Siblings unaffected

Other sessions on other GenServers never knew anything happened. Their heaps, their mailboxes, their state — all independent.

Code written: the happy path only. Recovery is structural, not behavioral. You don't predict failure modes — you define restart strategies.
1

LLM returns garbage JSON

Same trigger. Model hallucinates invalid syntax. This is normal — LLMs are non-deterministic.

2

try/catch intercepts

The error is caught. But now you need to decide: retry? fallback? log and continue? What state was mutated before the throw?

3

Error state management

Reset partial state. Clear buffers. Rebuild context. Check if the error escaped into shared structures (event emitters, callback chains, Map entries).

4

Fallback + retry logic

Exponential backoff. Fallback response. Maybe a different model. Each failure path is hand-written application code.

5

Hope nothing leaked

All sessions share the event loop and the V8 heap. An uncaught exception in a native callback can crash the entire process. Effect mitigates this, but the boundary is behavioral.

Code written: every failure path, manually. Works when you can enumerate failures. With non-deterministic agents, you can't — and the happy path disappears under error handling.
